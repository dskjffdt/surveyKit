## 一、简历项目栏（推荐 · 分模块写法，直接复制）

**SurveyKit | 在线问卷系统**  
独立开发 · 2025.XX – 2026.XX  
GitHub：https://github.com/dskjffdt/surveyKit · 在线演示：你的 Demo 地址

**项目概述：**  
基于 Web 的在线问卷系统，支持问卷创建、编辑、发布、公开填答与结果分析，涵盖单选、多选、填空、评分四种题型，适用于问卷收集、活动调研和反馈回收等场景。前端基于 React + TypeScript 构建，使用 Zustand 管理业务状态、React Router 拆分 SPA 页面，封装 Axios 对接 REST 接口并完成登录鉴权，已实现 Vite 构建与线上部署。

**技术栈：** React、TypeScript、Zustand、React Router、Axios、ECharts、Vite

**项目亮点：**

- **多题型类型建模**：基于 TypeScript 联合类型定义四种题型及差异化答案结构，通过类型收窄统一编辑、填答、统计三端处理逻辑，校验与聚合统计复用同一套类型体系，保证多题型场景下的类型安全与可维护性。

- **题型渲染与编辑交互**：封装题目渲染组件，实现编辑配置与公开填答双模式复用，避免各题型在编辑区与填答区重复开发；支持拖拽调整题目顺序，配合 Zustand 管理问卷编辑状态并在内容变更后同步持久化，提升问卷配置过程中的操作效率与交互一致性。

- **问卷状态流转**：设计草稿与已发布状态切换方案，按题型差异实现分层校验策略，覆盖必填校验、防重复提交、未保存离开提醒等关键节点，完整支持问卷从创建配置、公开发布到答卷回收的全流程状态控制。

- **权限与页面路由**：基于 React Router 实现管理端路由守卫，将登录鉴权、编辑管理与公开填答访问路径分离；封装 Axios 请求层统一对接后端接口，结合多角色权限控制区分管理员与创建者操作范围，满足多人协作场景下的访问隔离与安全校验。

- **数据统计与投放**：基于 ECharts 实现按题型的选项分布与评分统计可视化，统计模块按需加载以降低首屏加载成本；支持 CSV 数据导出、填答链接一键复制，并在首页汇总答卷回收情况，形成从问卷设计、发布投放到结果分析的完整闭环。

**项目成果：**

- 独立完成问卷配置 → 发布投放 → 答卷回收 → 统计分析 → 数据导出全流程闭环
- 支持 4 种题型、2 种角色权限，覆盖列表、编辑、填答、统计 4 个核心业务页面
- 前端已完成 Vercel 部署，统计与分享模块按需加载，支持完整在线演示

## 四、项目亮点（面试官问「有什么亮点」）

我觉得有三个比较实在的亮点：

**第一，业务闭环完整。** 不是只做编辑器或填答页，而是从创建、发布、填答到统计导出全部打通，能完整演示。

**第二，类型设计有针对性。** 四种题型答案结构不同，我用联合类型把编辑、填答、统计三处逻辑统一起来，避免后期维护时类型混乱。

**第三，编辑与投放体验比较完整。** 包括拖拽排序、双模式渲染复用、ECharts 图表统计、填答链接分享，不只是基础 CRUD 页面。

## 五、难点回答（必背）

难点主要在两个地方：

**第一个是「多题型场景下的数据一致性」。** 同一个题型会同时出现在编辑、填答、统计三个环节，答案类型各不相同。我用联合类型和分支收窄把三处逻辑统一起来，渲染层用双模式复用避免重复开发。

**第二个是「业务状态流转的完整性」。** 比如发布态控制、校验、重复提交、未保存提醒，这些逻辑串起来，项目才真正可用。

## 六、常见追问速答

**为什么用 Zustand？**  
状态规模适中，Zustand 写法简单，配合 API 拉取数据够用，比 Redux 轻。

**为什么 JWT 放 Cookie 而不是 localStorage？**  
httpOnly Cookie 前端 JS 读不到，能降低 XSS 偷 token 的风险；面试口述即可，简历写「登录鉴权」。

**admin 和 creator 权限区别？**  
admin 能看和管理全部问卷；creator 只能 CRUD 自己创建的。前后端都做了过滤。

**ECharts 怎么用的？**  
单选用饼图，多选和评分用柱状图，统计页按需加载，不拖慢首页。

**拖拽排序怎么实现的？**  
题目左侧拖拽手柄，松手后更新题目数组顺序并保存到后端。

**演示账号？**  
admin / admin123（管理员），creator / creator123（创建者）。
